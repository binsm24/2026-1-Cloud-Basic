## 1-1. AWS 글로벌 인프라

### AWS 글로벌 인프라

ASW 글로벌 인프라 : 우수한 품질의 글로벌 네트워크

### AWS 리전

AWS 리전 : 지리적 영역

- 리전 전체의 데이터 복제는 고객이 제어
- 리전 간 통신에는 AWS 백본 네트워크 인프라가 사용

각 AWS 리전은 완전한 이중화 및 네트워크 연결 제공

리전은 일반적으로 2개 이상의 사용 영역으로 구성

### 리전 선택

서비스, 애플리케이션, 데이터에 적합한 리전을 결정할 때 고려할 요인

1. 데이터 거버넌스, 법적 요구 사항
2. 고객에 대한 접근성
3. 리전 내에서 사용 가능한 서비스
4. 비용

### 가용 영역

각 리전에는 다수의 가용 영역이 존재

가용 영역 :  AWS 인프라의 완전히 격리된 파티션, 데이터 센터의 묶음

### AWS 데이터 센터

AWS 데이터 센터는 보안을 고려하여 설계

데이터 센터 : 데이터가 상주하고 데이터가 처리되는 위치

### 상호 접속 위치 (PoP:Points of Presence)

AWS는 상호 접속 위치(PoP) 로케이션으로 구성된 글로벌 네트워크 제공

엣지 로케이션과 훨씬 적은 수의 리전 엣지 캐시로 구성됨

Amazon CloudFront와 함께 사용됨

- Amazon CloudFront : 짧은 지연 시간으로 최종 사용자에게 콘텐츠를 전송하는 글로벌 콘텐츠 전송 네트워크(CDN)

### AWS POP & Edge

POP : AWS의 콘텐츠 전공 및 네트워크 서비스를 지원하는 물리적 데이터 센터

트래픽 허브 역할을 하며 글로벌 네트워크와 리전 데이터 센터를 연결하는 역할

### AWS 인프라 기능

1. 탄력성 및 확장성
    - 탄력적 인프라, 동적 용량 조정
    - 확장 가능한 인프라, 증가량 수용을 위해 조정
2. 내결함성
    - 장애 발생 시에도 계속 정상 운영
    - 구성 요소의 내장된 중복성
3. 고가용성
    - 높은 수준의 운영 성능
    - 가동 중단 시간 최소화
    - 인적 개입X

## 2-1. 네트워크 기초

### 네트워크

네트워크 : 서로 연결되어 리소스를 공유하는 2개 이상의 클라이언트 시스템

- 서브넷으로 논리적으로 분할 가능
- 클라이언트 간에 통신하려면 네트워킹 디바이스가 필요

### 공인 IP vs 사설 IP

공인 IP : 인터넷 사용자의 로컬 네트워크를 식별하기 위해 ISP가 제공하는 IP 주소

→ 외부에 공개되어 있는 IP 주소로 전세계에서 유일한 IP 주소

사설 IP : 일반 가정이나 회사 내 등에 할당된 네트워크의 IP 주소

→ 라우터에 의해 로컬 네트워크 상의 PC나 장치에 할당됨 

## 2-2. Amazon VPC

### Amazon VPC

Amazon Virtual Private Cloud(Amazon VPC) : 논리적으로 격리된 공간을 프로비저닝하여 AWS 리소스를 구동할 수 있는 고객이 정의한 가상 네트워크

### VPC 배포

모든 AWS 리전에서 VPC를 배포할 수 있음

VPC는 리전 내 모든 가용 영역의 지원되는 리소스를 호스팅할 수 있음

### Amazon VPC 기능

1. 가상 네트워킹 리소스 제어
- IP 주소 범위 선택
- 서브넷 생성
- 라우팅 테이블 및 네트워크 게이트웨이 구성
1. VPC에 대한 네트워크 구성을 사용자 지정 가능
2. 다중 보안 계층을 사용하여 각 서브넷에서 인스턴스에 대한 액세스를 제어

### Amazon VPC 할당량

기본 할당량 : 리전별 계정단 VPC 5개

### VPC 및 서브넷

VPC : 집

- 리전을 벗어날 수 X

서브넷 : 방, VPC를 분할하는 IP 주소의 범위

- 가용 영역을 벗어날 수 X → 단일 가용 영역에 속함
- 퍼블릭 또는 프라이빗으로 분류

### IP 주소 지정

VPC를 생성할 때 IPv4 CIDR 블록에 VPC 할당

### 퍼블릭 IP 주소 유형

퍼블릭 IPv4 주소 할당 방식 : 서브넷 수준에서 퍼블릭 IP 주소 자동 할당 설정을 통해 자동으로 할당, 탄력적 IP 주소를 통해 수동으로 할당

탄력적 IP 주소 : AWS 계정과 연결, 동일한 인스턴스, 다른 인스턴스에 언제든지 할당하고 다시 매핑할 수 있음

### VPC 설계 모범 사례

1. 한 리전의 모든 가용 영역에 걸쳐 VPC 네트워크 범위를 균등하게 나눔
2. 모든 네트워크 주소를 한 번에 할당X but 나중에 사용할 수 있도록 일부 주소 공간 예약
3. 예상되는 큰 폭의 워크로드 증가를 지원하도록 VPC CIDR 및 서브넷 크기 조정
4. VPC 네트워크 범위가 조직의 다른 사설 네트워크 범위와 중첩되지 않는지 확인

### 단일 VPC와 다중 VPC

단일 VPC 배포 : 제한적인 사용 사례에서는 하나의 VPC 배포가 적절

- 소규모 팀이 관리하는 소규모 단일 애플리케이션
- 고성능 컴퓨팅
- 자격 증명 관리

다중 VPC가 적합한 경우

1. 단일 팀 또는 단일 조직
2. 더 쉽게 표준 상태를 유지하고 액세스를 관리할 수 있는 제한된 팀

복수계정이 적합한 경우

1. 여러 팀에서 관리하는 애플리케이션을 배포하는 엔터프라이즈 고객 또는 조직
2. 빠른 성장이 예상되는 중간 규모의 조직
